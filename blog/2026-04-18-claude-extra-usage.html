Your Claude Subscription Has Three Wallets (and Opus Charges the Wrong One)

Claude 订阅其实是三个钱包（Opus 却刷错了那个）

Apr 18, 2026
Claude Debugging Agents

My Claude Code dropped mid-task with out of extra usage. My subscription dashboard said I'd used 11%. These two facts seem to contradict each other — until you realize Claude's subscription is actually three separate wallets, and I'd been pouring money into the wrong one.

TL;DR

Claude.ai subscription quota is split into three non-interchangeable pools:

今天 Claude Code 干到一半崩了，错误消息是 out of extra usage。我看了一眼订阅面板，主池才用了 11%。这两件事看起来矛盾——直到我意识到，Claude 订阅其实是三个独立钱包，而我一直在把钱烧在错的那一个上。

TL;DR

Claude.ai 订阅的额度是三个互不相通的池子：

Pool My status
All models (subscription)11% used
Sonnet only0% used
Extra usage (overflow wallet)101% used — stuck here

Opus and 1M-context runs draw from Extra usage. I had Claude Code and a 24/7 self-hosted agent (Hermes) both defaulting to Opus, quietly burning through the overflow wallet while my subscription sat untouched.

池子 我的状态
All models（订阅主池）用了 11%
Sonnet only用了 0%
Extra usage（超额钱包）用了 101% — 卡在这

Opus 模型和 1M context 都走 Extra usage。我的 Claude Code 和一个常驻自建 Agent（Hermes）默认都是 Opus，悄悄把超额钱包烧穿了，订阅主池却几乎没动。

The "running but dead" agent

I run a persistent agent on a Mac Mini called Hermes — 5 profiles, talking to me through Telegram and Discord. One morning I noticed it hadn't responded in 14 hours. But:

  • ps showed the gateway process still running
  • gateway_state.json said state: running, both Telegram and Discord connected

Every heartbeat indicator looked healthy. Send a message — silence.

Layer 1: Claude Code was stuck too

When I opened Claude Code to debug Hermes, it failed immediately:

"在运行，但是死了"的 Agent

我在 Mac Mini 上跑了一个叫 Hermes 的常驻 Agent——5 个 profile，通过 Telegram 和 Discord 收发消息。某天早上我发现它 14 小时没回消息，但：

  • ps 能看到 gateway 进程还活着
  • gateway_state.json 显示 state: running，Telegram 和 Discord 都是 connected

所有心跳指标都显示正常。但发消息——没反应。

第一层：Claude Code 自己也挂了

想打开 Claude Code 帮我排查 Hermes，结果它自己先报错了： 

Error code: 400 - out of extra usage.
Add more at claude.ai/settings/usage

I was confused. My subscription month wasn't over. How could I be out of anything?

The Claude dashboard gave the answer:

我懵了。订阅月还没结束，怎么会"超额"？

登录 Claude 面板才看清楚： 

All models        Resets Thu 3:00 PM    11% used
Sonnet only       Resets Fri 4:00 PM     0% used
Extra usage       Resets May 1         101% used   ← $20.17 spent

Two things I didn't know:

Opus burns fast and overflows into Extra usage when its subscription allowance runs out — the bar for overflow is much lower than I assumed.

The [1M context] variant bypasses the subscription entirely — it draws straight from Extra usage, regardless of how empty your main pool is.

I spend most of my day in claude-opus-4-7[1m]. That combination was quietly siphoning the overflow wallet while the main subscription pool sat at 11%.

Fix for layer 1: /model → Sonnet 4.6. Sonnet's dedicated pool was 0% used. Problem gone.

Layer 2: Hermes defaults to Opus too

While investigating, I found that ~/.hermes/config.yaml had model.default: claude-opus-4-6 — and each of the 5 profile configs (profiles/*/config.yaml) overrode it back to Opus. So Claude Code was burning Opus on the interactive side, and my always-on Hermes was burning Opus in the background. Both pouring into the same overflowed wallet.

Batch-switched everything to Sonnet:

两个我之前不知道的事：

Opus 消耗特别快，订阅额度一旦用完就会溢出到 Extra usage——而这个溢出门槛比我想象的低很多。

[1M context] 版本会直接绕过订阅——不管你订阅主池还剩多少，它都从 Extra usage 里扣。

我大部分时间都在用 claude-opus-4-7[1m]。这个组合就在我不知情的情况下，把超额钱包一点点烧光了，而订阅主池还停在 11%。

第一层的解法：/model → 切到 Sonnet 4.6。Sonnet 专属池子 0% used，瞬间满血。

第二层：Hermes 也默认用 Opus

排查过程中发现，~/.hermes/config.yaml 里默认就是 model.default: claude-opus-4-6——而且 5 个 profile 的 profiles/*/config.yaml 里各自又覆盖回了 Opus。也就是说：Claude Code 在交互端烧 Opus，后台常驻的 Hermes 也在烧 Opus，两边一起把同一个钱包吃干净。

批量改成 Sonnet： 

# global
~/.hermes/config.yaml  →  model.default: claude-sonnet-4-6

# 5 profiles
for p in main career diet scholar social; do
  sed -i.bak 's/default: claude-opus-4-6/default: claude-sonnet-4-6/' \
    ~/.hermes/profiles/$p/config.yaml
done

Then hermes gateway restart.

Layer 3: OAuth credential locked in "exhausted"

The gateway came back up clean, but every message I sent hit:

然后 hermes gateway restart。

第三层：OAuth 凭据被锁成"exhausted"

Gateway 重启后看起来正常，但只要一发消息就报错： 

Provider authentication failed: No Anthropic credentials found.

I had logged in via OAuth before (claude /login). Inspecting ~/.hermes/auth.json:

之前明明用 claude /login 走过 OAuth。翻 ~/.hermes/auth.json 看到： 

{
  "active_provider": null,
  "credential_pool": {
    "anthropic": [{
      "last_status": "exhausted",
      "expires_at_ms": 1776518463033
    }]
  }
}

Two overlapping problems:

  1. Access token expired 6.6 hours ago (expires_at_ms was 396 minutes in the past).
  2. last_status: "exhausted" — when Opus drained the Extra usage wallet, Hermes marked the whole credential as exhausted and refused to retry it.

hermes auth reset anthropic cleared the flag, but as soon as a message came in, the expired access token failed a real request and the flag came right back. Refresh silently wasn't recovering it.

The fix that worked: re-run OAuth from scratch.

两个问题叠加：

  1. Access token 已经过期 6.6 小时（expires_at_ms 比当前时间早 396 分钟）。
  2. last_status: "exhausted" — 当 Opus 把 Extra usage 烧爆时，Hermes 把这个凭据标记为"耗尽"并拒绝再用。

试了 hermes auth reset anthropic 清状态。清了，但一发消息这个过期的 access token 请求失败后又立刻被标回 exhausted。Refresh 也默默没救活它。

最终管用的办法：重新走一次 OAuth。 

hermes auth add anthropic --type oauth
# opens browser → log in → paste auth code
# → Added anthropic OAuth credential #2

hermes auth list showed the new credential marked active (←). Sent a message on Discord — Elsa came back.

Takeaways

For anyone running self-hosted agents on a Claude subscription instead of raw API keys:

  1. Subscription ≠ one pool. It's at least three — All models, Sonnet only, and Extra usage. Opening Opus or a 1M-context variant pulls from Extra usage, so the overflow wallet can empty before your subscription dashboard looks stressed.
  2. Background agents default to Opus. Hermes, Claude Code, many hobby agents ship with Opus as the default. On OAuth/subscription access, that means the most expensive pool gets charged first. If Sonnet is enough for the task, set it explicitly.
  3. OAuth credentials have an "exhausted" lock. After quota recovers, the client doesn't automatically unlock. You need hermes auth reset <provider> — and if the access token has also expired, a full hermes auth add re-login.
  4. "Process alive" ≠ "service healthy." ps and state files showed running for 14 hours of silence. Always check business-log timestamps, not just the process watchdog.

hermes auth list 确认新凭据激活（←）。去 Discord 发消息——Elsa 活了。

几个教训

写给跟我一样拿订阅自建 Agent 的人：

  1. 订阅 ≠ 一个池子。至少是三个——All models、Sonnet only、Extra usage。随手开 Opus 或 1M context 都会走 Extra usage，所以超额钱包可能先烧光，订阅面板看起来却还很宽裕。
  2. 后台 Agent 默认就是 Opus。Hermes、Claude Code、很多自建 Agent 的默认模型都是 Opus。走 OAuth/订阅时，等于优先把最贵的池子花掉。Sonnet 够用的场景就显式设成 Sonnet。
  3. OAuth 凭据有"exhausted"锁。额度恢复后客户端不会自动解锁。要手动 hermes auth reset <provider>——如果 access token 也同时过期了，那就得 hermes auth add 重新登录一次。
  4. "进程在跑" ≠ "服务健康"。ps 和 state 文件显示 running 了 14 个小时的沉默。永远看业务日志的时间戳，不要只看进程看门狗。
